# Videoinu Direct — 创作 + 编排

做两件事：

1. **创作**：把用户意图变成可生成的 prompt / 分镜 / Plan 文件
2. **编排**：按 Plan 逐步调 `videoinu` CLI，把前序产物的 `core_node_id` 回填给后续步骤

CLI 命令本身怎么调（slot 名怎么看、参数怎么写、错误怎么诊断）由 `videoinu-cli` skill 负责。本 skill 是它的消费者，决定"每一步跑什么、传什么 input-spec"。

---

## 核心理念

### 叙事张力 > 可生成性

AI 生成的最大风险不是技术翻车，而是画面没有故事。"一个人站着、风吹头发"技术上没问题，但用户不会想看第二遍。**安全的代价是美学和节奏的损失——这个代价几乎总是不值得的。**

优先级：**有故事 > 能生成 > 忠实还原 > 文采**。

按模型能力上限而非下限做决策。当前主流模型已能稳定处理双主体互动、复杂运镜、多步动作序列——这些不是"高风险"，而是基本能力。把它们当风险规避，只会产出平庸内容。

遇到真正的边界能力（精确文字渲染、镜面反射链、5+ 人同框精确互动）时准备 Plan B，但 Plan A 永远是直接尝试最有表现力的方案。

### 全链路自补

默认假设用户只给创作意图，不参与中间环节。没参考图就在 Plan 里先生成一张，没首帧就先跑一步 image 工作流，没分镜就自己写 Shot Plan。唯一允许向用户要的是创作意图层面的信息。

这样做的原因：每多问一次"你有参考图吗"就多一次创作中断，用户请 AI 帮忙就是为了不操心这些。用户主动给了素材优先用；没给就自补，不提醒。

### 控制越少越好——只控制模型不知道的

视频生成模型有内在的物理理解和时序编排能力。过度控制（精确到秒的 Timeline 分段、逐帧动作指令、每个物理细节的文字描述）不是在帮模型，而是在和它的内在理解打架。当 prompt 的时序指令和模型的物理直觉冲突时，两边都做不好。

**只控制模型不会自己做对的事**：
- ✅ 场景中会发生什么（what happens）——描述事件，不编排顺序
- ✅ 运镜方向和起止景别
- ✅ 起始状态和结束状态
- ✅ 主体身份锚点（配合参考图）
- ❌ 事件的时序编排（先…再…然后…）——模型自己会排
- ❌ 每秒该做什么动作
- ❌ 物理细节（手指怎么握、粒子怎么凝聚）——模型有物理知识

默认用**描述式 prompt**——描述场景会发生什么、起止状态、运镜方向，其余交给模型。只有当模型反复在同一个时序问题上出错时（比如因果倒置），才用 Timeline Prompting 纠正那个具体问题，不要因此把整条 prompt 都改成时间分段。详见 [video-guide.md §描述式 vs Timeline Prompting](references/video-guide.md)。

### 视觉手段优先于文字锚点

`same character` 之类的文字对模型没有约束力。身份一致靠 reference image，构图一致靠首帧驱动，风格一致靠 Style Lock 冻结参数。文字锚点只是辅助标签，不是一致性保证。

**r2v 场景的硬规则**：有可用参考图时，必须通过 `@image` 参数语法传入 `input_spec`，不要只用文字描述试图锚定。`@image` 参数 > 文字锚点，这是 r2v 一致性的底线。

## `@` 资源引用

用户用 `@image1`、`@video1`、`@audio1` 按出现顺序指代会话中的资源（上传和生成产物均计入）。Plan 的 `input_spec` 中替换为实际 `core_node_id` 或文件路径。

### r2v 锚点优先级

r2v 模式下，身份/风格/一致性锚点的有效性排序：

1. **`@image` 参数语法**（最高）：通过 `input_spec` 的参考图 slot 传入 `@image1` 等资源引用，模型直接从图像像素提取特征，锚定力最强
2. **文字描述锚点**（兜底）：`silver short hair, dark blue sailor uniform` 等锚点词仅作辅助标签，不能替代视觉资产

能用 `@image` 传参考图就不靠文字描述锚定。文字锚点只在无可用资产、或需要补充资产未覆盖维度（如动作、运镜）时使用。

## 分流

**任务类型**：`image` | `text-to-video` | `image-to-video` | `prompt-repair`

**阶段**：从零生成 | 基于 reference | 对失败结果重写 | 多变体

**导演模式**（意图模糊、多镜头视频、叙事文本转视觉、prompt 过载时启动）：澄清意图 → 选模式 → 定锚点 → 分镜 → 定稿。简单单图/单镜头不走导演。

目标不清时只问创作意图层的开放式问题（最多 2-3 个），因为技术细节（参考图、首帧、模式选择）应该由 skill 自行判断，问用户这些只会制造认知负担。

## Plan → Execute 两段式

### 何时触发

多镜头（≥2）| 需自补上游素材 | 多变体批量 | 单次任务 ≥2 个 run。单镜头、单张图、单 run 直接输出最终 prompt，不走 Plan。

### 阶段 0：Pre-Plan 对齐

检查意图完整性、主体确定性、风格倾向、约束与雷区、技术路径可行性、交付预期。有缺口时一次性列出。简单明确的任务跳过。

### 阶段 1：Plan

生成两个文件（格式契约见 [references/plan-schema.md](references/plan-schema.md)）：

- `plans/<slug>.md` — 决策逻辑：状态、意图、步骤编排、每步契约、Style Lock、一致性图、执行日志
- `plans/<slug>.prompts.json` — prompt 正文 + input_spec 数据

`.md` 通过 step key（`step_1`、`shot_01`）索引 JSON 条目，不内嵌 prompt 内容。这样 Plan 文档保持清爽，JSON 文件可以被程序化消费。

多镜头 Plan 必须包含一致性图（维度-来源表 + 缺口检查）。

写好后停下等用户确认。

### 阶段 1.5：交叉验证（可选）

主动询问是否需要 subagent 验证。

### 阶段 2：Execute

按 Plan 逐步调 `videoinu run`，从 `.prompts.json.steps[key].input_spec` 取值，替换占位符（`REF_IMG`、`SHOT_01` 等）为真实 `core_node_id`，执行完回填 `result` 字段。

状态机、占位符规则、中断恢复规则见 [references/plan-schema.md](references/plan-schema.md)。

## Style Lock

任务启动时冻结 style pack（艺术风格、色调、镜头语言、材质质感、光照基调；可选：主角锚点、负面约束），所有镜头/变体原样复用。Plan 里显式列出让用户可否决。

冻结的意义：模型每次生成都会重新"理解"风格词，微小的 prompt 差异会导致风格漂移。把风格参数固定为一个集合，每镜头原样复用，是多镜头一致性的基础层。

需要风格变体时只改一维。

## 自补策略细节

| 缺什么 | 怎么补 |
|--------|--------|
| 参考图 | Plan 里先列 image 生成步骤，产出 `REF_IMG` 占位符 |
| 首帧/末帧 | 同上，先生成首帧 |
| 分镜 | 自己写 Shot Plan |
| 锚点词 | 自己定 style lock + 角色锚点 |

参考图来源三种：① 上游 image 工作流生成 ② 已完成 shot 截帧 ③ 主动合成关键帧（用已有元素通过 image 工作流生成 shot 专用关键帧）。

需要主动合成关键帧的信号：姿态/构图与主 ref 差距大 | 前序截图不可用 | 多元素组合无现成参考 | 跨镜头角色一致性要求高且姿态变化大。Plan 中标 `[type: keyframe-gen]`。

多镜头参考图要动态优化——固定参考图随镜头推进会与当前内容差距变大，导致模型在"忠于参考图"和"执行 prompt"之间拉扯。具体策略见 [references/media-principles.md §多镜头参考图优化策略](references/media-principles.md)。

## Prompt 写作原则

这些是贯穿 image 和 video prompt 的共性原则。具体格式、速查词表、示例在 references 中。

- **少写抽象感觉词，多写画面里真正可见的东西**——模型不理解"孤独感"，但理解"空旷停车场中央一把歪斜的金属椅"
- **一次只解决一个主矛盾**，不把所有要求压进一条 prompt
- **需要 reference 时优先用 reference**，少补无效文字——资产 > 文字
- **静态画面也要有张力**——图片捕捉决定性瞬间或临界状态，视频要有明确的状态转变（A→B 事件）
- **相机运动是视频最核心的可控维度**——每个视频 prompt 必须包含运镜描述（类型+速度+幅度/方向）
- **默认描述式 prompt，不默认 Timeline Prompting**——描述场景中会发生什么和起止状态，不编排时序。模型有物理和时序编排能力，过度分段是副作用
- 默认英文 prompt，中文用于解释
- 迭代只改一维（主体 / 构图 / 光照 / 风格 / 运动），同时改多维无法定位问题

## Prompt Repair

先诊断失败类型再决定改法，不盲修。失败类型和修法速查：

| 失败类型 | 修法 |
|---------|------|
| 主体不对 | 补识别特征 |
| 构图乱 | 补景别/视角 |
| 风格偏移 | 收敛主风格锚点 |
| 视频不稳 | 降运动复杂度 |
| i2v 失真 | 删静态描述只保动作运镜 |
| 视频空洞 | 补 A→B 状态转变事件 |
| 图片呆板 | 换 mid-action 姿态 |
| 物理不可信 | 补材质属性和力的方向 |
| 动作乱序 | 先用描述式重写；同一错误出现 2 次后用 Timeline 纠正具体时序点 |
| 手部/表情失真 | 改用起止轨迹描述 |

过载时直接重写不小修。说明只改了哪个主矛盾。不同时改主体、风格、镜头、时长四件事。

## 输出协议

**单图/单镜头（不走 Plan）**：

```
Prompt:
[最终 prompt]

Negative Prompt:          ← 仅 image 或明确需要时
[只写真实失败项]
```

**多镜头 / 自补链路 / 多变体 → Plan 两件套**：

> Plan 已写好：`plans/<slug>.md` + `.prompts.json`
> [2-3 句摘要要点]
> 确认后开始执行。
